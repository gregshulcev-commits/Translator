# Нейронный перевод в проекте

## Короткий вывод

Для этого проекта нейронный перевод остаётся **optional layer**, а не частью ядра.

Это решение уже реализовано в v6:

- базовый словарный режим остаётся лёгким и полностью рабочим без нейронки;
- контекстный нейронный перевод подключается отдельно;
- офлайн-модель ставится по требованию через GUI или CLI;
- пользователь сам решает, нужны ли ему дополнительные сотни мегабайт на модель.

## Почему это правильный баланс

### Что важно для ядра

Ядро приложения должно быстро:

- открыть документ;
- найти слово;
- показать перевод по словарю;
- не зависеть от сети;
- не требовать гигабайтных паков по умолчанию.

### Что важно для нейронки

Нейронный слой полезен, когда нужно перевести **контекст предложения**, а не только отдельное слово. Но это обычно означает:

- заметно более тяжёлый runtime;
- большие модели;
- отдельную установку;
- потенциально больший расход RAM и времени на первый запуск.

## Что уже сделано в v6

### Argos как первый offline neural prototype

В проекте реализован optional Argos-сценарий:

- `ArgosContextProvider` для перевода предложения;
- `argos_manager.py` для проверки runtime и моделей;
- GUI manager **Перевод → Офлайн-модели Argos…**;
- установка модели из сети;
- импорт локального `.argosmodel` файла;
- CLI helper `tools/install_argos_model.py --list / --file / --from-lang / --to-lang`.

### Что это даёт пользователю

- ядро не разрастается до тяжёлого дистрибутива;
- можно один раз скачать модель и дальше работать полностью офлайн;
- можно заранее получить `.argosmodel` на другой машине и импортировать его локально;
- базовый словарный режим остаётся быстрым даже без нейронки.

## Практический ресурсный вывод

Для EN↔RU лучше считать нейронку именно **устанавливаемым пакетом**, а не обязательной частью приложения.

Поэтому для проекта разумно сохранять правило:

1. словарный перевод слова — базовый режим;
2. контекстный перевод предложения — optional provider;
3. офлайн-модель — отдельная установка;
4. по умолчанию нейронка выключена.

## Следующий реалистичный шаг

После стабилизации Argos-ветки можно развивать не ядро, а именно optional-пакетный слой:

- удаление/обновление моделей из GUI;
- дополнительные offline-направления;
- file-level neural translation как отдельный сервис;
- сравнение с более тяжёлыми runtime только после подтверждения реальной пользы.
