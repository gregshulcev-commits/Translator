==================================================
1. ЧТО ЭТО ЗА ПРОЕКТ
==================================================

Это offline-first переводчик документов по клику на слово.

На текущем этапе проект состоит из двух связанных веток:

1. desktop MVP на Python + Tkinter для Linux;
2. отдельная Android/APK branch prototype с нативным Android UI и переиспользованием Python-слоя словаря через bridge.

Главная идея продукта:

- открыть документ;
- кликнуть/выбрать незнакомое слово;
- получить словарную подсказку;
- при необходимости показать контекстный перевод предложения через optional provider layer;
- не превращать словарный режим в тяжёлый monolith.

Поддерживаемые форматы desktop на текущем этапе:

- PDF
- TXT
- FB2

Направления:

- EN -> RU
- RU -> EN

Desktop умеет:

- просмотр документа;
- scroll и zoom;
- поиск по документу;
- click-to-translate;
- compact bottom translation panel;
- словари как SQLite packs;
- optional provider layer: Disabled / Argos / LibreTranslate / Yandex Cloud;
- GUI install/import словарей;
- GUI install/import Argos offline models;
- zoom до 800% — НЕ УМЕНЬШАТЬ верхний предел без явной причины, пользователь попросил оставить большой zoom.

Android branch пока не является готовым продуктом, но в кодовой базе уже есть:
- `android-client/`
- `src/pdf_word_translator/mobile_api.py`
- Kotlin UI prototype
- Chaquopy bridge
- PdfRenderer prototype
- bundled starter SQLite dictionaries as Android assets

3. ЧТО НЕЛЬЗЯ ЛОМАТЬ
==================================================

1. Нельзя превращать проект в “только нейронный переводчик”.
   Словарный перевод слова остаётся базовым и обязательным сценарием.

2. Нельзя жёстко встраивать нейронную модель в ядро lookup.
   Контекстный/нейронный перевод должен оставаться отдельным optional provider layer.

3. Нельзя снова начинать спор “встраивать модель в ядро или нет”.
   Ответ уже принят:
   - не встраивать жёстко;
   - устанавливать модели отдельно;
   - держать provider port и отдельный lifecycle.

4. Нельзя показывать пользователю длинное сырое предложение из документа как основной UI-элемент нижней панели.
   Hidden context допустим как внутренний input для provider layer, но не как “сырая простыня текста” в compact panel.

5. Нельзя пытаться тянуть Tkinter на Android.
   Android = отдельный клиент.

6. Нельзя откатывать исправление вычисления PROJECT_ROOT:
   `PROJECT_ROOT = PACKAGE_ROOT.parent.parent`

7. Нельзя без необходимости ломать текущую терминологию:
   - direction
   - dictionary pack
   - provider
   - context translation
   - mobile bridge
   - desktop reference implementation

8. Не создавать отдельные release notes-файлы, если это не требуется явно.
   Все изменения и инструкции по установке должны отражаться в `README.md` и соответствующих `.md` в `docs/` и тематических подкаталогах документации.

5. КЛЮЧЕВОЙ РАБОЧИЙ ПОТОК DESKTOP
==================================================

5.1. Словарный перевод по клику
- пользователь кликает по canvas;
- `MainWindow` переводит координаты в page coordinates;
- `TranslationWorkflow.translate_point()` ищет token через `DocumentSession.find_token_at()`;
- `DictionaryService.lookup()` делает словарный lookup;
- нижняя panel обновляется;
- highlight рисуется поверх текущей страницы;
- при активном provider layer запускается контекстный перевод.

5.2. Контекстный перевод
Контекстный перевод НЕ встроен в dictionary lookup.
Он работает отдельным слоем через `ContextTranslationProvider`.

Смысл:
- словарный перевод слова и перевод предложения — это разные задачи;
- словарный режим должен работать даже без нейронки и без сети;
- provider ошибки не должны ломать основной сценарий.

5.3. Hidden context semantics
- sentence/context извлекается в workflow;
- UI не обязан показывать полный текст пользователю;
- этот hidden context используется как input для provider;
- результат provider попадает во вторую строку compact panel;
- длинный raw text документа не должен становиться видимой простынёй.

7. ЧТО БЫЛО СДЕЛАНО В V5
==================================================

Исправления v5, которые нельзя терять:

1. Исправлен scroll-back bug:
   раньше при выделении слова lazy redraw возвращал viewport обратно к выделенному слову.
   Теперь обычная перерисовка highlight не должна насильно scroll-ить к нему.

2. Исправлен баг Treeview rowheight:
   при увеличении UI font size названия словарей вылезали за пределы строки.
   Теперь row height масштабируется вместе с font metrics.

3. Исправлена доставка async provider result:
   результат больше не должен прилетать в Tk напрямую небезопасным образом.
   Используется очередь и обработка в main thread.

4. Усилена устойчивость `settings.json`:
   лишние ключи и битые значения не должны ронять приложение на загрузке.

5. Уточнена и усилена валидация `Folder ID` для Yandex Cloud.

Это уже часть новой базовой стабильности. Не ломать.

==================================================
11. СЛАБЫЕ МЕСТА И ИЗВЕСТНЫЕ ОГРАНИЧЕНИЯ
==================================================

11.1. Viewer / performance
- lazy rendering уже есть, но на тяжёлых PDF и больших zoom всё ещё могут быть фризы;
- нет background render queue / prefetch jobs;
- нет thumbnails;
- нет overview/sidebar страниц.

11.2. Text interaction
- desktop PDF viewer работает через render в canvas, а не как полноценный native selectable text shell;
- нет полноценного системного выделения текста как в нативном PDF reader;
- нет полного copy-to-clipboard сценария для выделенного текста;
- click-to-translate есть, но это не то же самое, что selectable PDF shell.

11.3. Dictionaries
- качество перевода зависит от состава и порядка packs;
- нет удаления словарей из GUI;
- нет reorder/prioritize GUI;
- нет явной диагностики, какой pack дал конкретный translation hit.

11.4. Provider layer
- Argos manager уже есть, но lifecycle можно ещё улучшать;
- LibreTranslate / Yandex не должны становиться обязательными;
- online providers не должны ломать словарный режим;
- вторая строка compact panel семантически должна оставаться аккуратной.

11.5. Formats
- нет OCR;
- нет DOCX/EPUB;
- нет текста на изображениях.

11.6. Android
- tap-to-word selection ещё отсутствует;
- lookup пока по ручному вводу слова;
- provider layer ещё не интегрирован в Android UI;
- Android build pipeline ещё не финализирован.
