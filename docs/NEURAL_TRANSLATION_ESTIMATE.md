# Оценка ресурсов для нейронного перевода

## Короткий вывод

Для этого проекта нейронный перевод лучше рассматривать как **optional plugin / language pack**, а не как часть ядра.

Причины:

- модели EN↔RU занимают сотни мегабайт каждая;
- для двух направлений диск легко выходит за 1 ГБ;
- на слабых устройствах RAM и latency становятся заметными.

## Практический план для текущего проекта

### Сейчас

- словарный перевод остаётся базовым режимом;
- контекстный перевод вынесен в отдельный provider layer;
- один из провайдеров может быть офлайн-нейронным (`Argos`).

### Что это даёт

- ядро приложения остаётся лёгким;
- пользователь сам решает, ставить ли тяжёлую модель;
- Linux-архив не разрастается до гигабайтов;
- Android не блокирует desktop-релиз.

## Оценка по ресурсам

### Argos

Практически удобен как first neural prototype, потому что:

- ставится отдельно;
- модель хранится как `.argosmodel` пакет;
- интегрируется Python-скриптом без перепаковки всего приложения.

### Более тяжёлые Marian / OPUS-подобные модели

Если переходить к отдельной sentence-level модели с самостоятельным runtime, безопасно закладывать порядок величин:

- сотни мегабайт на одно направление;
- около гигабайта и больше за двунаправленный набор с runtime-накладными расходами;
- заметное потребление RAM при инференсе.

## Рекомендация

Лучший путь для этого проекта:

1. ядро остаётся словарным;
2. нейронный перевод добавляется как **plugin layer**;
3. модель ставится отдельно через GUI / installer script;
4. по умолчанию нейронка выключена.
