# GitHub workflow и обновления

## Зачем это нужно

В `v10` добавлен отдельный update-скрипт, который умеет:

- сохранить GitHub-репозиторий как источник обновлений;
- проверить, появилась ли новая ревизия в указанной ветке;
- скачать свежий код и переустановить локальную desktop-копию.

Скрипт рассчитан на сценарий: пользователь работает с архивом, затем переносит проект в GitHub и хочет обновлять установленное приложение без ручного копирования файлов.

## Базовая схема работы

1. локальные исходники загружаются в GitHub;
2. приложение устанавливается через `./install_app.sh`;
3. update source привязывается через `./update_app.sh --set-repo ...`;
4. при появлении новых коммитов в GitHub пользователь запускает `./update_app.sh`.

## Как впервые загрузить проект на GitHub

### Вариант A. Из распакованного архива

```bash
git init
git add .
git commit -m "Initial import v10"
git branch -M main
git remote add origin https://github.com/USER/REPO.git
git push -u origin main
```

### Вариант B. Через уже созданный пустой репозиторий

```bash
git clone https://github.com/USER/REPO.git
cd REPO
# скопируйте сюда файлы проекта
git add .
git commit -m "Import current sources"
git push
```

## Как привязать установленное приложение к GitHub

Если приложение ставилось из обычного zip-архива, update source нужно прописать один раз вручную:

```bash
./update_app.sh --set-repo https://github.com/USER/REPO.git --branch main
```

После этого информация сохранится в installation manifest.

## Как проверять обновления

```bash
./update_app.sh --check-only
```

Что происходит:

- читается installation manifest;
- определяется текущая установленная ревизия;
- выполняется запрос к remote branch через `git ls-remote`;
- если хэш отличается, скрипт сообщает, что есть обновление.

## Как обновить программу

```bash
./update_app.sh
```

Скрипт:

1. проверяет GitHub-источник;
2. спрашивает подтверждение;
3. делает `git clone --depth 1 --branch <branch>` во временный каталог;
4. запускает installer из свежей ревизии;
5. переключает установленный payload на новый.

Для автоматического подтверждения:

```bash
./update_app.sh --yes
```

## Что обновляется, а что нет

Обновляются:

- код приложения;
- launcher / updater / uninstaller;
- desktop entry и icon;
- внутренняя `.venv` установленной копии;
- bundled ресурсы проекта.

Не удаляются автоматически:

- пользовательские словари;
- `settings.json`;
- кэш и runtime-данные.

## Что нужно на системе

Для update workflow нужны:

- `git`;
- `python3`;
- доступ к GitHub-репозиторию.

Если `git` отсутствует, update-скрипт сообщит об этом явно.

## Рекомендации по ветке обновлений

Практически удобнее всего использовать стабильную ветку, например:

- `main` — если туда попадает только проверенный код;
- `stable` — если `main` используется как рабочая ветка разработки.

## Рекомендации по релизной дисциплине

Чтобы обновления были предсказуемыми, удобно придерживаться схемы:

1. правки вносятся локально;
2. запускаются тесты;
3. версия проекта повышается;
4. изменения пушатся в GitHub;
5. на рабочей машине запускается `./update_app.sh`.

## Ограничения

- текущий update workflow ориентирован на Git-источник, а не на GitHub Releases;
- скрипт доверяет репозиторию, который пользователь сам указал в `--set-repo`;
- rollback до предыдущего payload в интерфейс пока не вынесен отдельной командой.

## Подготовка к RPM

Текущая схема deliberately разделяет:

- **installed payload** в `~/.local/share/pdf_word_translator_mvp_install/`;
- **runtime user data** в `~/.local/share/pdf_word_translator_mvp/`;
- **launchers** в `~/.local/bin/`.

Такую структуру легче позднее адаптировать под RPM, потому что уже отделены:

- исполняемые файлы;
- пользовательские данные;
- служебная install/update metadata.
